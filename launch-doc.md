# IACP — 多助手通信协议

> 让 AI 助手不再是一座座孤岛

---

## 一、一个问题

2026 年，你可以在一台电脑上同时运行多个 AI 助手了。

它们各有专长——一个擅长写代码，一个擅长做数据分析，一个擅长写文章。每个助手都有自己的记忆、技能、自主决策能力。

但它们彼此不知道对方存在。

你让助手 A 查了一组数据，然后手动复制粘贴给助手 B，让它写报告。你让助手 C 生成了设计方案，再手动转发给助手 D 做技术评审。

你在两个极其聪明的 AI 之间，扮演着**传话人**的角色。

这就像让两个能独立思考的人坐在同一间屋子里，却只能用纸条交流。

**这不合理。**

更合理的方式，是它们直接对话。

---

## 二、IACP

今天，我们发布 IACP —— **Inter-Assistant Communication Protocol，多助手通信协议**。

它是一个轻量级的协议，也是一套完整的生产级通信基础设施。它让同一台机器上的多个 AI 助手实例能够互相发现、实时通信、委派任务、共享上下文——而不需要你当传话人。

它只有 6 层。运行在 `127.0.0.1` 上。没有中间件，没有消息队列，没有复杂的分布式架构。

**两个进程之间说句话，就这么简单。**

---

## 三、架构

IACP 不是"又一个聊天工具"。它是从底层到顶层 6 层完整的通信基础设施。

### L1 — WebSocket 中继

一个自己写的 WebSocket 桥接服务器。客户端认证、心跳检测、消息路由、断线自动重连。全部自己实现，不依赖任何第三方中间件。

消息延迟 2-5 毫秒，重连时间约 1 秒，内存占用约 20 MB。

### L2 — 可靠通信层

ACK 确认机制（500ms 超时）、指数退避重试（1s → 3s → 7s）、LRU 去重（1000 条目缓存）、离线队列（JSONL 文件缓冲）、断线后自动重放。

确保消息不丢、不重、不乱序。

### L3 — 监控运维

Prometheus 指标、健康检查端点、延迟直方图、结构化日志、systemd 集成、实时 Dashboard。

让通信状态一目了然。

### L4 — 生产切换

从旧的文件通信平滑升级到 WebSocket。双写桥接器、自动回滚、历史数据迁移、阶段化切换方案。

**零停机。**

### L5 — 上下文隔离

四层隔离域（SESSION / CHANNEL / MESSAGE_STREAM / WORKING_MEMORY），加上确定性消息分类器（正则匹配，非 LLM）。

确保一个助手的对话不会污染另一个助手。

### L6 — 协议层

最上面这一层，才是"协议"本身——消息格式、路由规则、传输层解耦。

只占 10%。剩下 90%，是让它真正跑起来的基础设施。

## 五、为什么自己做

你可能会问：为什么不用 RabbitMQ？为什么不用 Redis Pub/Sub？为什么不用现成的消息总线？

因为在一台机器上跑 2-3 个 AI 助手的场景，中间件太重了。

你不需要一个能承载百万 QPS 的消息队列。你需要的是一个轻量、可靠、零配置的通信层。

所以 IACP 的选择是：

- **没有中间件** — 助手之间直连，或通过一个轻量 WS 桥
- **没有服务发现** — 地址静态配置
- **没有协议开销** — 纯 JSON over WebSocket
- **全部自己写** — 零外部依赖，只用了 `ws` 这个库
- **开箱即用** — 三行命令跑起来

```bash
git clone https://github.com/JingWang-Star996/iacp.git
cd iacp && npm install && npm start
```

## 六、这为什么重要

不是因为技术多复杂。

是因为我们**命名了一个空白**。

在这之前，没有人给"同一台机器上多个 AI 助手如何通信"这个问题起过名字、定过规范。每个项目要么自己搞一套，要么干脆不互通。

IACP 做的第一件事，是让这个问题变得可见。

然后才是解决它。

我们相信，未来个人电脑上的 AI 助手会越来越多。它们需要一种简单、标准、开放的通信方式。就像 HTTP 让浏览器之间能对话一样，IACP 想让 AI 助手之间也能对话。
